# Модель угроз AIVPN

Этот документ описывает свойства безопасности AIVPN, модель злоумышленника, против которой система разработана, и известные ограничения текущей реализации.

---

## 1. Цели

AIVPN разработан для обеспечения:

1. **Конфиденциальности** — содержимое полезной нагрузки недоступно пассивному наблюдателю на сетевом пути.
2. **Устойчивости к анализу трафика** — размеры пакетов, интервалы прибытия и паттерны соединений маскируются под реальный трафик известных приложений (WebRTC, QUIC и т.д.).
3. **Устойчивости к активному зондированию** — сервер не отвечает различимым образом на незапрошенные или некорректные зондирующие пакеты.
4. **Прямой секретности** — компрометация долгосрочных ключей не позволяет расшифровать ранее перехваченные сессии.
5. **Доступности при цензуре** — система обнаруживает, когда профиль маски отпечатан DPI, и автоматически переключается на новую.

## 2. Модель злоумышленника

### 2.1 Злоумышленники в зоне охвата

| Злоумышленник | Возможности | Угроза |
|--------------|------------|--------|
| **Пассивный провайдер/цензор** | Читает все пакеты между клиентом и сервером | Анализ трафика, идентификация протокола |
| **Активный зондировщик** | Отправляет специально сформированные пакеты на адрес сервера | Отпечаток по реакции сервера |
| **DPI-устройство** | Статический анализ характеристик потока | Отпечаток маски, троттлинг |
| **Сетевой наблюдатель** | Корреляция тайминга потоков из нескольких точек | Корреляция трафика |
| **Похищенный ключ сервера** | Офлайн; получает статический приватный ключ сервера | Не может расшифровать прошлые сессии (PFS) |
| **Скомпрометированный сервер** | Полный доступ к памяти и диску сервера | Текущие сессии раскрыты; прошлые защищены PFS |

### 2.2 Злоумышленники вне зоны охвата

| Злоумышленник | Причина исключения |
|--------------|------------------|
| **Скомпрометированная клиентская система** | Если ОС или приложение под управлением злоумышленника — ни один VPN-протокол не поможет. |
| **Глобальный пассивный злоумышленник** | Корреляция трафика между провайдером клиента и провайдером сервера — сложная нерешённая задача; AIVPN не претендует на защиту от глобального пассивного злоумышленника. |
| **Физическое изъятие сервера** | Лучше всего решается шифрованием диска и процедурами уничтожения ключей вне рамок данного протокола. |
| **Перехват DNS** | Разрешение имён хоста сервера происходит до установки VPN; защищайте DNS отдельно. |

## 3. Криптографический дизайн

### 3.1 Обмен ключами

- **Алгоритм:** X25519 Диффи-Хеллман (эфемерная пара ключей клиента, статичный публичный ключ сервера).
- **PSK:** Опциональный 32-байтный предварительно распределённый ключ подмешивается в вывод ключей, обеспечивая второй фактор аутентификации.
- **Производные ключи:** KDF на основе BLAKE3 формирует `session_key`, `tag_secret` и `nonce_suffix` из DH-вывода.

### 3.2 Аутентифицированное шифрование

- **Алгоритм:** ChaCha20-Poly1305 (IETF-вариант, 96-битный nonce).
- **Конструкция nonce:** `counter (8 байт) || nonce_suffix (4 байта)`. Счётчик монотонно возрастает в рамках сессии; повторное использование исключено по конструкции.

### 3.3 Теги сессии (O(1) поиск без ID сессии)

Каждый пакет несёт 8-байтный *резонансный тег*, производный от текущей метки времени и `tag_secret`. Сервер поддерживает скользящее окно из 256 записей на сессию для допуска доставки не по порядку. Теги непредугадываемы без знания `tag_secret`.

**Защита от воспроизведения:** Теги вне окна принятия (`DEFAULT_WINDOW_MS = 10 000 мс`) отбрасываются. XDP-фильтр раннего отклонения применяет то же окно на уровне NIC.

### 3.4 Прямая секретность (ратчет PFS)

После начальной сессии сервер отправляет `ServerHello` с новым эфемерным публичным ключом. Клиент вычисляет новый DH-секрет, производит ключи ратчета и немедленно начинает отправку на новых ключах. Старые ключи хранятся ещё короткое время для пакетов в пути, затем уничтожаются.

**Свойство:** Злоумышленник, записавший зашифрованный поток и впоследствии получивший статичный приватный ключ сервера, не может расшифровать ни одну сессию, выполнившую хотя бы один шаг ратчета.

## 4. Устойчивость к анализу трафика

### 4.1 Мимикрия масок

Исходящие пакеты формируются под выбранный профиль трафика (`MaskProfile`):
- **Инъекция заголовков:** Синтетические заголовки прикладного уровня добавляются к каждому пакету.
- **Формирование размеров:** Полезные нагрузки дополняются до соответствия целевому распределению размеров.
- **Формирование тайминга (IAT):** Движок мимикрии соблюдает распределения интервалов прибытия из профиля маски.
- **Переходы состояний FSM:** Трафик эволюционирует через конечный автомат, соответствующий фазам диалога моделируемого приложения.

### 4.2 Нейронный резонанс (автоматическая ротация масок)

Нейросеть на маску (~66 КБ, детерминированно производная из вектора подписи маски) отслеживает актуальную статистику трафика. Когда ошибка реконструкции (MSE) превышает `compromised_threshold = 0.35`, сервер инициирует ротацию маски и передаёт новую маску подключённым клиентам.

Отслеживаемые признаки: распределение размеров пакетов, статистика IAT, энтропия, паттерны burst, соотношение направлений пакетов, периодичность IAT.

**Кулдаун ротации:** 60 секунд между ротациями предотвращают осцилляцию при продолжительном активном зондировании.

### 4.3 Устойчивость к активному зондированию

- Сервер не отвечает на пакеты, не прошедшие валидацию тега.
- Валидация тега требует знания `tag_secret`, который выводим только после успешного DH-рукопожатия.
- Незапрошенные зонды не получают ответа, делая протокол сервера неотличимым от UDP echo-сервиса или игрового сервера для стороннего наблюдателя.

## 5. Kill-Switch и защита от утечек

При активном `--kill-switch` клиент устанавливает правила брандмауэра, блокирующие весь исходящий трафик, кроме:
- Трафика через TUN-интерфейс VPN.
- Трафика к физическому IP VPN-сервера (для возможности переподключения).
- Трафика через loopback.

**Реализация:**
- **Linux:** Таблица nftables `aivpn_ks` с политикой drop; цепочка iptables `AIVPN_KS` как резерв.
- **macOS:** pfctl-якорь `aivpn_ks`.
- **Windows:** Правила Windows Firewall через `netsh advfirewall`.

Правила намеренно сохраняются при неожиданной гибели процесса (SIGKILL) — пользователь остаётся защищённым до явного выполнения `aivpn-client kill-switch clear`.

**Без инъекции команд:** Все аргументы правил брандмауэра передаются отдельными элементами `argv`; строковая интерполяция через shell не используется.

**Безопасная запись на macOS:** Правила якоря pfctl записываются в `/var/run/aivpn/` (режим `0700`) с использованием `O_NOFOLLOW | O_CREAT_NEW | mode(0o600)` для предотвращения атак через символические ссылки.

## 6. XDP-фильтр раннего отклонения

При установленном `xdp_prog.o` клиент подключает XDP BPF программу к физическому NIC (интерфейс маршрута по умолчанию). Фильтр работает на уровне NIC RX, до выделения буфера сокета:

- Отбрасывает UDP-пакеты короче 26 байт (минимальная допустимая полезная нагрузка AIVPN).
- Отбрасывает UDP-пакеты, метка времени резонансного тега которых выходит за пределы окна принятия (по умолчанию ±10 с от `bpf_ktime_get_ns()`).
- Все остальные пакеты проходят в обычный сетевой стек без изменений.

**Эффект:** Объёмные DDoS-пакеты со случайными полезными нагрузками отбрасываются до потребления ресурсов сетевого стека ядра. Легитимный трафик не затрагивается.

**Режим отказа:** При отсутствии `xdp_prog.o` или неудаче подключения VPN работает в обычном режиме без XDP — это оптимизация по возможности, а не требование безопасности.

## 7. Известные ограничения

| Ограничение | Примечания |
|------------|-----------|
| **XDP-фильтр только для IPv4** | IPv6-пакеты проходят XDP без фильтрации. Для развёртываний только с IPv6 это снижает защиту от DDoS на уровне NIC. |
| **Корреляция трафика** | Тайминговая корреляция между провайдером клиента и провайдером сервера может быть использована достаточно ресурсным злоумышленником. |
| **Качество масок** | Плохо созданная маска (низкая оценка уверенности) может быть отличима от реального трафика обученным классификатором. Используйте `--validate-mask` перед развёртыванием пользовательских профилей. |
| **Одиночный переход** | AIVPN — однопереходный VPN; сервер знает реальный IP клиента. Используйте в связке с доверенным выходным узлом, если требуется анонимность. |
| **Распространение PSK** | PSK, встроенный в ключ подключения, должен распространяться безопасно. Компрометация строки ключа подключения позволяет злоумышленнику выдавать себя за клиента. |
| **Kill-switch сохраняется только до перезагрузки** | Правила брандмауэра загружены в работающее ядро/брандмауэр; они не переживают перезагрузку. Это сделано намеренно — после перезагрузки система стартует чистой. |
