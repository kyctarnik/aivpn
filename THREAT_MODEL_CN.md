# AIVPN 威胁模型

本文档描述 AIVPN 的安全属性、系统所针对的攻击者模型，以及当前实现的已知局限性。

---

## 1. 安全目标

AIVPN 旨在提供：

1. **机密性** — 网络路径上的被动观察者无法获取载荷内容。
2. **流量分析抵抗** — 数据包大小、到达间隔和连接模式被伪装成已知应用程序的正常流量（WebRTC、QUIC 等）。
3. **主动探测抵抗** — 服务器不会对未经请求或格式错误的探测数据包作出可识别的响应。
4. **前向安全性** — 长期密钥的泄露不会导致之前捕获的会话被解密。
5. **抗审查可用性** — 系统能检测到伪装配置文件被 DPI 指纹识别的情况，并自动切换到新配置。

---

## 2. 攻击者模型

### 2.1 范围内的攻击者

| 攻击者 | 能力 | 威胁 |
|--------|------|------|
| **被动 ISP/审查机构** | 读取客户端与服务器之间的所有数据包 | 流量分析、协议识别 |
| **主动探测者** | 向服务器端点发送精心构造的数据包 | 通过服务器响应行为进行指纹识别 |
| **DPI 设备** | 对流量特征进行状态分析 | 伪装配置文件指纹识别、限速 |
| **网络观察者** | 跨多个观测点关联流量时序 | 流量关联分析 |
| **服务器密钥被窃** | 离线获取服务器静态私钥 | 无法解密过去的会话（PFS 保护） |
| **服务器被入侵** | 完全访问服务器内存和磁盘 | 当前会话暴露；历史会话受 PFS 保护 |

### 2.2 范围外的攻击者

| 攻击者 | 排除原因 |
|--------|---------|
| **客户端端点被入侵** | 若攻击者控制了运行客户端的操作系统或应用程序，任何 VPN 协议都无法提供保护。 |
| **全局被动攻击者** | 客户端 ISP 和服务器 ISP 之间的流量关联是一个难题；AIVPN 不声称能抵抗全局被动攻击者。 |
| **服务器物理没收** | 最好通过全盘加密和密钥销毁程序来处理，这超出了本协议的范围。 |
| **DNS 拦截** | 服务器主机名解析发生在 VPN 建立之前；请单独保护 DNS。 |

## 3. 密码学设计

### 3.1 密钥交换

- **算法：** X25519 Diffie-Hellman（临时客户端密钥对、静态服务器公钥）。
- **PSK：** 可选的 32 字节预共享密钥混入密钥派生过程，提供第二认证因素。
- **派生密钥：** 基于 BLAKE3 的 KDF 从 DH 输出派生 `session_key`、`tag_secret` 和 `nonce_suffix`。

### 3.2 认证加密

- **算法：** ChaCha20-Poly1305（IETF 变体，96 位 nonce）。
- **Nonce 构造：** `counter (8 字节) || nonce_suffix (4 字节)`。计数器在会话内单调递增；设计上防止重用。

### 3.3 会话标签（无会话 ID 的 O(1) 查找）

每个数据包携带一个 8 字节的*共振标签*，由当前时间戳和 `tag_secret` 派生。服务器为每个会话维护一个 256 条目的滑动窗口，以允许乱序传递。没有 `tag_secret` 的情况下，标签是不可猜测的。

**防重放：** 接受窗口外的标签（`DEFAULT_WINDOW_MS = 10 000 毫秒`）将被丢弃。XDP 早期过滤器在 NIC 层面执行相同的窗口检查。

### 3.4 完美前向安全（PFS 棘轮）

初始会话后，服务器发送携带新临时公钥的 `ServerHello`。客户端计算新的 DH 秘密，派生棘轮密钥，并立即开始使用新密钥发送数据。旧密钥在短暂保留以处理在途数据包后即被销毁。

**安全属性：** 攻击者即使录制了密文流并在事后获取了服务器静态私钥，也无法解密任何已完成至少一个棘轮步骤的会话。

## 4. 流量分析抵抗

### 4.1 伪装配置文件模拟

出站数据包按选定的流量配置文件（`MaskProfile`）进行整形：
- **头部注入：** 向每个数据包添加合成应用层头部。
- **大小整形：** 对载荷进行填充以匹配目标大小分布。
- **时序整形（IAT）：** 模拟引擎执行配置文件中的到达间隔分布。
- **FSM 状态转换：** 流量通过有限状态机演进，匹配目标应用程序的对话阶段。

### 4.2 神经共振（自动配置文件轮换）

每个配置文件对应一个神经网络（约 66 KB，由配置文件签名向量确定性派生），实时监控流量统计数据。当重建误差（MSE）超过 `compromised_threshold = 0.35` 时，服务器触发配置文件轮换，并将新配置推送给已连接的客户端。

监控特征：数据包大小分布、IAT 统计、熵值、突发模式、数据包方向比、IAT 周期性。

**轮换冷却时间：** 两次轮换之间 60 秒的冷却时间，防止在持续主动探测下出现振荡。

### 4.3 主动探测抵抗

- 服务器不响应标签验证失败的数据包。
- 标签验证需要知道 `tag_secret`，而后者只能在成功完成 DH 握手后才能派生。
- 未经请求的探测不会收到响应，使服务器协议对外部观察者而言与 UDP echo 服务或游戏服务器无法区分。

## 5. Kill-Switch 与防泄漏保护

当 `--kill-switch` 处于活动状态时，客户端安装防火墙规则，阻止所有出站流量，仅允许：
- 通过 VPN TUN 接口的流量。
- 到物理 VPN 服务器 IP 的流量（以便重新建立隧道）。
- 回环流量。

**实现方式：**
- **Linux：** 带 drop 策略的 nftables 表 `aivpn_ks`；iptables 链 `AIVPN_KS` 作为备用。
- **macOS：** pfctl 锚点 `aivpn_ks`。
- **Windows：** 通过 `netsh advfirewall` 配置 Windows 防火墙规则。

规则在进程意外终止（SIGKILL）后有意保持有效——用户持续受到保护，直到显式运行 `aivpn-client kill-switch clear`。

**无命令注入：** 所有防火墙规则参数作为独立的 `argv` 元素传递；不通过 shell 进行字符串插值。

**macOS 安全写入：** pfctl 锚点规则写入 `/var/run/aivpn/`（模式 `0700`），使用 `O_NOFOLLOW | O_CREAT_NEW | mode(0o600)` 防止针对全局可写目录的符号链接攻击。

## 6. XDP 早期数据包过滤

当安装了 `xdp_prog.o` 时，客户端将 XDP BPF 程序附加到物理 NIC（默认路由接口）。过滤器在 NIC RX 层面运行，在套接字缓冲区分配之前：

- 丢弃短于 26 字节的 UDP 数据包（AIVPN 最小有效载荷）。
- 丢弃共振标签时间戳超出接受窗口（默认距 `bpf_ktime_get_ns()` ±10 秒）的 UDP 数据包。
- 所有其他数据包不经修改直接传递到正常网络栈。

**效果：** 具有随机载荷的大流量 DDoS 数据包在消耗内核网络资源之前就被丢弃。合法流量不受影响。

**故障模式：** 如果 `xdp_prog.o` 不存在或附加失败，VPN 照常运行而不使用 XDP——这是尽力而为的优化，而非安全要求。

## 7. 已知局限性

| 局限性 | 说明 |
|--------|------|
| **XDP 过滤器仅支持 IPv4** | IPv6 数据包无条件通过 XDP。对于纯 IPv6 部署，这会降低 NIC 层面的 DDoS 防护能力。 |
| **流量关联** | 客户端 ISP 和服务器 ISP 之间的时序关联可能被资源充足的攻击者利用。 |
| **伪装配置文件质量** | 制作粗糙的配置文件（置信度分数低）可能被训练有素的分类器与真实流量区分开来。部署自定义配置文件前请使用 `--validate-mask` 验证。 |
| **单跳架构** | AIVPN 是单跳 VPN；服务器知道客户端的真实 IP。如需匿名性，请结合可信出口节点使用。 |
| **PSK 分发** | 嵌入连接密钥中的 PSK 必须安全分发。连接密钥字符串泄露将允许攻击者冒充客户端。 |
| **Kill-switch 仅持续到重启** | 防火墙规则加载在运行中的内核/防火墙中；重启后不保留。这是有意为之——重启后系统以干净状态启动。 |
